# nanoTracker Plugin — LLM Reference Guide
# Version: 1.0  |  Schema: v1  |  Updated: 2026-04-08
#
# This file is authoritative context for language models generating nanoTracker plugins.
# Read it fully before writing any plugin.json or script.js content.
# ──────────────────────────────────────────────────────────────────────────────


## OVERVIEW

nanoTracker plugins are ZIP archives with a custom extension:

  .ntins  — Instrument plugin (polyphonic voices, triggered by tracker notes)
  .ntsfx  — FX plugin (audio effect, slots into FX Mixer channels)

The archive contains:
  plugin.json   REQUIRED — all configuration, schema, and preset data
  script.js     OPTIONAL — AudioWorklet processor (runs in audio thread only)
  samples/      OPTIONAL — WAV or FLAC audio files referenced by plugin.json
  assets/       OPTIONAL — PNG/SVG graphics

Use build_plugin.py in this directory to package a source folder into an archive.


## ARCHITECTURE RULE — CRITICAL

Plugin-specific musical content (presets, pad patterns, pitch sequences, rhythm ideas)
MUST live inside plugin.json under "loopPresets[]" or the "dsp" block.

NEVER expect the host application source files to be modified for a specific plugin.
The host is generic. It reads plugin.json dynamically at runtime.

The host only provides:
  - Generic algorithmic presets (FORWARD, REVERSE — work for any N-pad instrument)
  - Schema parsing and rendering
  - UI infrastructure (step grid, FX mixer, parameter controls)

If you find yourself wanting the host code to know about a specific instrument
(hardcoded pad names, hardcoded preset patterns, instrument-specific UI logic),
put it in plugin.json instead.


## plugin.json — TOP-LEVEL STRUCTURE

{
  "schemaVersion": 1,          // integer, must be 1
  "manifest": { ... },         // required
  "parameters": [ ... ],       // optional, array of PluginParamDef
  "dsp": { ... },              // required
  "ui": { ... },               // optional
  "loopPresets": [ ... ]       // optional, instrument plugins only
}


## MANIFEST

"manifest": {
  "name":        "My Plugin",       // string, max 64 chars
  "version":     "1.0.0",           // string, semver recommended
  "type":        "instrument",      // "instrument" | "fx"
  "author":      "Author Name",     // string, optional
  "description": "One-liner"        // string, optional
}

The host uses name + version to form the plugin ID: "plugin:Name@version"
This ID is stored in .ftrk save files to reference the plugin.


## PARAMETERS

"parameters": [
  {
    "key":             "envelope.attack",  // dot-path for instruments, "nodeId.param" for fx
    "label":           "ATTACK",           // ALL-CAPS display label
    "min":             0.001,
    "max":             4.0,
    "default":         0.01,
    "step":            0.001,
    "unit":            "s",               // optional: "s", "Hz", "%", "dB", "ms", ""
    "displayDecimals": 3                  // optional integer, controls LCD display
  }
]

Parameter keys for FX plugins use "nodeId.audioParam" dot-notation.
  Examples: "wet.gain", "predelay.delayTime", "filter.frequency", "filter.Q"

Parameter keys for instrument plugins use dot-paths that the host resolves
against the dsp block. Common paths:
  "envelope.attack"    "envelope.decay"    "envelope.sustain"    "envelope.release"
  "filter.frequency"   "filter.Q"

Wet/dry parameters should use 0–100 range (host divides by 100 internally).
Time parameters use seconds as the unit.


## INSTRUMENT DSP BLOCK

"dsp": {
  "processorName": null,         // string | null — AudioWorklet processor name
  "voices":        16,           // integer 1–32
  "voiceStealing": "oldest",     // "oldest" | "quietest" | "none"

  "oscillators": [               // optional, mixed before samples
    { "type": "sawtooth", "detune": -10, "mix": 0.5 },
    { "type": "sawtooth", "detune":  10, "mix": 0.5 }
  ],

  "samples": [ ... ],            // see SAMPLE ZONES section

  "envelope": {
    "attack":  0.005,            // seconds
    "decay":   0.1,              // seconds
    "sustain": 0.8,              // 0.0 – 1.0 (amplitude ratio)
    "release": 0.3               // seconds
  },

  "filter": {                    // optional, null to omit
    "type":      "lowpass",      // see FILTER TYPES
    "frequency": 8000,           // Hz
    "Q":         1.0
  }
}

Oscillator types: "sine" | "square" | "sawtooth" | "triangle" | "noise"

Filter types: "lowpass" | "highpass" | "bandpass" | "notch" | "allpass"
              "peaking" | "lowshelf" | "highshelf"

When processorName is null, the host uses its built-in polyphonic sample player
(AudioBufferSourceNode + ADSR gain ramps + BiquadFilterNode). This covers the
vast majority of instrument plugin needs.

When processorName is set, a script.js must be present. The host registers the
AudioWorklet and routes note events via MessagePort (see AUDIOWORKLET section).


## SAMPLE ZONES

"samples": [
  {
    "file":          "samples/kick.wav",    // path within archive
    "rootKey":       36,                    // MIDI note number (C2 = 36)
    "keyRange":      { "lo": 24, "hi": 48 },  // MIDI range this zone handles
    "velocityRange": { "lo":  0, "hi": 127 }, // velocity range
    "loop":          false,
    "loopStart":     0.0,                   // seconds into buffer
    "loopEnd":       0.0,                   // seconds (0 = end of buffer)
    "startOffset":   0.0,                   // seconds into buffer to begin playback
    "duration":      0.0                    // seconds to play (0 = play to end)
  }
]

startOffset and duration enable non-destructive sample slicing.
A single audio file can back multiple zones with different offsets.

Example — 32 equal slices from a 4-bar loop at 170 BPM:
  step_dur = 60 / 170 / 2   // eighth-note duration in seconds
  zone[i].startOffset = i * step_dur
  zone[i].duration    = step_dur
  zone[i].rootKey     = 36 + i
  zone[i].keyRange    = { lo: 36 + i, hi: 36 + i }

Multiple zones can cover the same MIDI range with different velocityRanges
for velocity-switching (e.g. soft/medium/hard hits).

The host selects the best-matching zone for each note event based on:
  1. keyRange contains the note
  2. velocityRange contains the velocity
  3. If multiple zones match, first one wins


## FX DSP BLOCK

"dsp": {
  "processorName": null,         // string | null
  "nodes": [
    { "id": "predelay", "type": "delay",    "maxDelay": 0.5, "delayTime": 0.02 },
    { "id": "verb",     "type": "convolver","impulse": "samples/hall.wav", "normalize": true },
    { "id": "wet",      "type": "gain",     "gain": 0.3 },
    { "id": "dry",      "type": "gain",     "gain": 1.0 }
  ],
  "connections": [
    { "from": "input",    "to": "predelay" },
    { "from": "input",    "to": "dry"      },
    { "from": "predelay", "to": "verb"     },
    { "from": "verb",     "to": "wet"      },
    { "from": "wet",      "to": "output"   },
    { "from": "dry",      "to": "output"   }
  ]
}

Reserved IDs: "input" and "output" — these are GainNode stubs created by the host.
Do NOT define nodes with these IDs.

FX Node Types and their settable fields:

  gain        gain
  delay       delayTime, maxDelay
  biquad      frequency, Q, gain, filterType (string: Web Audio BiquadFilterNode type)
  compressor  threshold, ratio, attack, release, knee
  convolver   impulse (file path in archive), normalize (boolean)
  panner      pan (-1.0 to 1.0)
  waveshaper  curve ("sigmoid" | "clip" | "fold"), drive
  worklet     (processor-defined AudioParams from parameterDescriptors)

Parameter keys in "parameters[]" for FX plugins must be "nodeId.audioParam":
  "wet.gain"           → nodes.find(n => n.id === "wet") → GainNode.gain
  "predelay.delayTime" → DelayNode.delayTime
  "filter.frequency"   → BiquadFilterNode.frequency
  "filter.Q"           → BiquadFilterNode.Q
  "comp.threshold"     → DynamicsCompressorNode.threshold


## LOOP PRESETS (instrument plugins only)

loopPresets define named step sequences that appear as one-click preset buttons
in the tracker's loop generator UI. All creative content for a specific instrument
MUST be here — never in host source code.

"loopPresets": [
  {
    "name": "CLASSIC",           // max ~12 chars for clean display
    "steps": [
      {
        "padIndex": 1,           // 1-based zone index (1 = zone[0]). 0 = silent step
        "pitch":    0,           // semitone offset, default 0
        "volume":   100,         // 0–100, default 100
        "reverse":  false,       // play sample reversed, default false
        "active":   true         // false = mute this step, default true
      },
      { "padIndex": 0 },         // silent step (minimal form)
      { "padIndex": 3, "volume": 75 }
    ]
  },
  {
    "name": "BREAKCORE",
    "steps": [ ... ]
  }
]

Steps array length determines the preset pattern length. Common lengths: 8, 16, 32.
The host renders each step at the current BPM, one step per subdivision.
Shorter step arrays loop within the current pattern length.


## UI BLOCK

"ui": {
  "layout": "grid",             // "grid" | "flex"
  "controls": [
    { "type": "knob",          "parameter": "envelope.attack",  "label": "ATTACK"  },
    { "type": "slider",        "parameter": "filter.frequency", "label": "CUTOFF"  },
    { "type": "toggle",        "parameter": "filter.enabled",   "label": "FILT"    },
    { "type": "number",        "parameter": "voices",           "label": "VOICES"  },
    { "type": "waveform_view", "sampleIndex": 0,                "label": "WAVE"    }
  ]
}

Control types:
  knob          Rotary dial (270° sweep). Best for continuous params.
  slider        Vertical or horizontal fader.
  toggle        ON/OFF button.
  select        Dropdown for discrete values.
  number        LCD click-to-edit numeric field.
  waveform_view Canvas waveform display. Requires "sampleIndex" (integer, 0-based zone index).

All numeric values displayed using the tracker's LCD ghost-digit style automatically.
Labels should be ALL-CAPS, ≤8 chars for best display.


## AUDIOWORKLET — script.js

Only needed for custom DSP that cannot be expressed as a declarative node graph.
Most plugins do NOT need script.js.

Security: script.js runs exclusively in the AudioWorklet scope (audio rendering thread).
No DOM, no fetch, no XMLHttpRequest. The host never eval()s it.

### FX Processor (script.js)

class MyFxProcessor extends AudioWorkletProcessor {
  static get parameterDescriptors() {
    return [
      { name: "drive", defaultValue: 0.5, minValue: 0, maxValue: 1, automationRate: "k-rate" }
    ];
  }

  process(inputs, outputs, parameters) {
    const input  = inputs[0];
    const output = outputs[0];
    const drive  = parameters.drive[0];
    for (let ch = 0; ch < output.length; ch++) {
      const inp = input[ch] ?? new Float32Array(128);
      const out = output[ch];
      for (let i = 0; i < out.length; i++) {
        out[i] = Math.tanh(inp[i] * (1 + drive * 10));
      }
    }
    return true;
  }
}
registerProcessor("my-fx-processor", MyFxProcessor);

The processorName in plugin.json must exactly match the string in registerProcessor().


### Instrument Processor (script.js)

Instrument processors receive note events via MessagePort. The host does NOT send
raw MIDI — it sends structured JS objects.

class MySynthProcessor extends AudioWorkletProcessor {
  constructor() {
    super();
    this._voices = [];
    this.port.onmessage = ({ data }) => {
      if (data.type === "noteOn") {
        // data: { type, note, velocity, frequency }
        this._voices.push({ freq: data.frequency, vel: data.velocity / 127, phase: 0 });
      } else if (data.type === "noteOff") {
        // data: { type, note }
        this._voices = this._voices.filter(v => v.note !== data.note);
      } else if (data.type === "allNotesOff") {
        this._voices = [];
      } else if (data.type === "param") {
        // data: { type, key, value }
        this._applyParam(data.key, data.value);
      }
    };
  }

  process(inputs, outputs) {
    const out = outputs[0][0];
    if (!out) return true;
    for (let i = 0; i < out.length; i++) {
      let s = 0;
      for (const v of this._voices) {
        s += Math.sin(2 * Math.PI * v.freq * v.phase / sampleRate) * v.vel;
        v.phase++;
      }
      out[i] = s / Math.max(1, this._voices.length);
    }
    return true;
  }
}
registerProcessor("my-synth-processor", MySynthProcessor);


### Message Protocol (Host → Processor)

Type          Payload
noteOn        { type: "noteOn",     note: int, velocity: int, frequency: float }
noteOff       { type: "noteOff",    note: int }
allNotesOff   { type: "allNotesOff" }
param         { type: "param",      key: string, value: number }

frequency is already converted from MIDI note number by the host. Use it directly
for oscillator frequencies. The conversion is:
  frequency = 440 * 2^((note - 69) / 12)

## BUILD TOOL USAGE

From this directory:

  python build_plugin.py ./my-plugin-source
    → reads my-plugin-source/plugin.json
    → validates all fields
    → packages into my-plugin-source/../my-plugin-source.ntins (or .ntsfx)

  python build_plugin.py ./my-plugin-source --out ./dist
    → writes archive to dist/

  python build_plugin.py ./my-plugin-source --name "PluginName_1.0.0"
    → overrides output filename stem

The builder validates:
  - schemaVersion is supported (currently: 1)
  - manifest has name, version, type
  - All parameters have min < max, default within range
  - Sample file paths exist in the source directory
  - FX connections reference valid node IDs
  - loopPresets steps have valid padIndex and volume values
  - processorName / script.js consistency

Warnings (non-fatal):
  - Default value outside [min, max] range
  - Sample file path not found (might be intentional for generated content)
  - processorName set but no script.js found
  - script.js present but processorName is null


## COMMON MISTAKES

1. Hardcoding plugin logic in the host
   WRONG: Expecting host code to special-case your plugin by name
   RIGHT: Put all creative content in plugin.json loopPresets[]

2. processorName mismatch
   WRONG: processorName: "my-synth" but script.js calls registerProcessor("MySynth", ...)
   RIGHT: The strings must be identical

3. Connections using undefined node IDs
   WRONG: { "from": "reverb", "to": "output" } when no node has id "reverb"
   RIGHT: Every id in connections must appear in nodes[], or be "input"/"output"

4. Sample file paths wrong
   WRONG: "file": "kick.wav" when the file is at samples/kick.wav in the archive
   RIGHT: "file": "samples/kick.wav" — paths are relative to archive root

5. sustain out of range
   WRONG: "sustain": 80   (host expects 0.0–1.0)
   RIGHT: "sustain": 0.8

6. padIndex 0-based vs 1-based
   loopPresets padIndex is 1-BASED. 0 = silent step. 1 = first sample zone.
   WRONG: { "padIndex": 0 } to mean the first zone
   RIGHT: { "padIndex": 1 } for zone[0], { "padIndex": 0 } for silence

7. Missing "input"/"output" connections
   Every FX plugin must have at least one connection from "input" and at least
   one connection to "output", otherwise no audio will pass through.
